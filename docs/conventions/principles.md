# Principles — KISS / YAGNI / DRY 그리고 SOLID

> 이 문서를 읽어야 할 때: 새 기능 설계 직전, 리팩터링 직전, 코드 리뷰 중 "이거 너무 복잡한데" 싶을 때.

---

## 0. 왜 이 원칙들이 필요한가

코드는 **작성되는 시간보다 읽히는 시간이 훨씬 길다**. 6개월 뒤의 자기 자신, 혹은 처음 이 저장소를 여는 기여자가 30초 안에 의도를 파악할 수 있어야 한다. 아래 원칙들은 "우아함을 과시하기 위한 학술 규칙"이 아니라, **미래의 읽는 사람의 인지 부하를 줄이기 위한 도구**이다.

이 프로젝트는 특히 **학습용 보일러플레이트**이므로, 복잡한 추상화·과도한 제네릭·투사되지 않은 미래를 위한 코드는 모두 "학습자의 이해를 가로막는 장벽"이다. 의심스러울 땐 단순한 쪽을 택하라.

---

## 1. KISS — Keep It Simple, Stupid

### 원리

가장 단순한 해결책이 항상 첫 후보이다. 단순함은 디버깅 비용, 리뷰 비용, 변경 비용, 학습 비용을 모두 낮춘다. 복잡도는 "지금 필요한 복잡도"만 허용한다.

### 왜 중요한가

- 복잡한 코드는 **작성 시점보다 3개월 뒤 디버깅할 때 3배 더 비싸다**.
- 단순한 코드는 **버그가 숨을 곳이 적다**.
- 팀원(또는 다른 AI 에이전트)이 동일한 멘탈 모델로 수정하기 쉽다.

### Go에서 KISS 적용하기

**Do**
- 인터페이스는 **두 번째 구현이 실제로 등장할 때** 추출한다. `UserRepository`가 Postgres 하나뿐이라면 구조체로 충분하다.
- `for ... range`로 될 일을 `reduce/map/filter`스러운 헬퍼로 포장하지 않는다. Go의 관용적 코드는 루프이다.
- 에러는 `if err != nil { return err }` 패턴 그대로 쓴다. 학습자가 흐름을 따라가기 쉽다.

**Don't**
- "혹시 모르니" 함수에 옵션 파라미터 6개 추가 (Options struct로 빼봐야 호출부가 복잡해진다).
- 아직 쓸 곳이 없는 제네릭 헬퍼 선언 (`func Map[T, U any](...) ...`는 진짜 3번 반복될 때 등장).
- 1개 구현체만 있는데 미리 mock 주입 가능한 인터페이스 추상화 (YAGNI와 겹친다).

**이 프로젝트의 실제 예시 (`backend/internal/`)**
- 현재 `Handler → Service → Repository` 3단 레이어 구조는 KISS의 상한선에 가깝다. 여기서 **더 쪼개지 말 것**. UseCase/DTO/Mapper 레이어를 추가하면 MVP 단계에서는 순수하게 오버엔지니어링이다.
- `cmd/server/main.go`에서 의존성 주입이 손으로 이루어지고 있는데(wire/fx 같은 DI 프레임워크 미사용), 이게 정답이다. 입문자가 "어디서 어떻게 연결되는지" 눈으로 따라갈 수 있다.

### React에서 KISS 적용하기

**Do**
- 상태는 가장 가까운 공통 부모에 둔다. "전역 상태 관리자가 필요한가?"는 prop drilling이 실제로 아플 때 묻는다.
- `useEffect`는 **외부와의 동기화가 필요할 때만** 쓴다. 버튼 클릭에 반응하는 건 effect가 아니라 이벤트 핸들러의 일이다.
- JSX가 길면 그때 컴포넌트를 추출한다. 미리 나누지 말라.

**Don't**
- 한 번 쓰고 말 훅을 `src/hooks/`로 빼지 말 것. 사용처 파일 안에 두는 게 낫다.
- Context를 "나중에 필요할 수 있으니" 미리 열지 말 것. Context는 **한 번 도입되면 제거가 어렵다**.

### KISS 체크리스트

- [ ] 더 단순한 대안을 **명시적으로** 검토했는가?
- [ ] 이 추상화는 *지금* 필요한가, *나중에 필요할 것 같은가*? (후자라면 YAGNI 위반)
- [ ] 입문자가 이 파일을 열고 30초 안에 의도를 파악할 수 있는가?
- [ ] 100줄이 넘는 함수/컴포넌트가 있다면, 정말 단일 책임인가?

## 2. YAGNI — You Aren't Gonna Need It

### 원리

**지금 필요하지 않은 기능은 지금 만들지 않는다.** 진짜 필요해질 때 만든다.

### 왜 중요한가

- "나중에 필요할 것 같은" 코드의 약 70%는 실제로 필요해지지 않거나, 필요해질 때쯤엔 요구사항이 달라져서 다시 만들게 된다.
- 만든 코드는 무료가 아니다 — **테스트, 문서, 유지보수, 리뷰 시간, 인지 부하** 모두 비용이다.
- "만들어 뒀으니 쓸 수 있다"는 sunk cost fallacy에 빠지면, 더 나은 대안을 선택하기 어려워진다.

### 금지 패턴 (이 프로젝트에서 실제로 유혹받기 쉬운 것들)

| 유혹 | 왜 YAGNI 위반인가 |
|------|-------------------|
| "나중에 MySQL/SQLite도 지원할지 모르니" Repository 인터페이스 추상화 | 현재 Postgres 전용으로 충분. 두 번째 DB가 요구사항이 되면 그때 추출한다. |
| "나중에 권한이 복잡해질지 모르니" RBAC 시스템 미리 구축 | 지금은 "인증됨/안 됨" 2개뿐. 역할이 추가될 때 그때 만든다. |
| "혹시 모르니" config 옵션 30개 노출 | 실제로 쓰는 건 DB 접속 정보 + JWT secret + server port(7개). 나머지는 컨텍스트 오염. |
| "i18n을 미리 해두는 게 낫지 않을까" | 한국어만 쓴다면 지금은 불필요. 실제 다국어 요구가 들어오면 그때 i18next 도입. |

### YAGNI의 예외 — 보안과 가드레일

**보안은 YAGNI의 예외이다.** 아래는 "필요해진 다음"에 추가하면 이미 늦다:
- 인증/인가 (JWT 검증, 비밀번호 해싱)
- 입력 검증 (SQL injection, XSS 방어)
- 시크릿 마스킹 (로그 누출 방지)
- HTTPS / Secure cookie 플래그 (prod 환경에서)

"나중에 붙이면 되지"가 성립하지 않는 영역이다. 이 프로젝트의 `backend/internal/service/auth_service.go`에서 bcrypt·JWT를 처음부터 적용한 것도 같은 이유이다.

### YAGNI 체크리스트

- [ ] 이 기능을 요구한 **현재** 사용자 스토리/이슈가 존재하는가?
- [ ] "나중에 X가 필요해질지도 모른다"라는 가정에만 기반하고 있지 않은가?
- [ ] 지금 만들지 않으면 나중에 **정말** 만들기 어려운가? (대부분은 아니다)

## 3. DRY — Don't Repeat Yourself

### 원리

**동일한 "지식"은 한 곳에서만 표현되어야 한다.** 단, "우연히 비슷해 보이는 코드"와 "본질적으로 같은 지식"은 구분해야 한다.

### 진짜 DRY가 가치 있는 것들

- 비밀번호 해싱 로직 (`bcrypt.GenerateFromPassword` 파라미터·cost factor는 한 곳)
- JWT 서명/검증 로직 (secret 사용처는 한 곳)
- DB 연결 설정 / DSN 조합 (`config.DSN()` 하나)
- 공통 에러 포맷 (API 응답의 `{"error": "..."}` 형태)

이런 것들은 실제로 동일한 결정 지식을 표현한다. 바뀔 때 모두 함께 바뀌어야 한다.

### 가짜 DRY — 섣부른 추출

다음은 **추출하지 말아야 할** 상황이다:

1. **두 핸들러가 우연히 비슷하게 생겼지만 한쪽이 곧 변경될 예정**
   - 추출하면 곧 "optional 파라미터"와 "분기"가 추가되어 더 복잡해진다.
2. **3번 반복되기 전까지는 추출하지 않는다 (Rule of Three)**
   - 2번 반복은 우연일 수 있다. 3번째가 나타나면 진짜 패턴이다.
3. **5줄 줄이려고 helper 함수로 빼는 경우**
   - 호출자가 helper를 읽으러 파일을 이동해야 한다. 총 비용은 오히려 증가한다.

### Go에서 DRY 판단하기

- 헬퍼 함수보다 **명시적 코드**가 종종 낫다. Go의 관용은 "반복을 두려워하지 않는 쪽"이다.
- 인터페이스 추출은 두 번째 구현이 **실제로 등장할 때**. 예상만으로 추출 금지.
- 구조체 임베딩은 "상속"이 아니라 "구성"이다. 코드를 줄이려고 임베딩하지 말고, 실제로 해당 동작을 구성 요소로 포함할 때만.

### React에서 DRY 판단하기

- 커스텀 훅은 상태 로직이 **진짜** 재사용될 때만. 한 곳에서만 쓰는 훅은 컴포넌트 안에 두어라.
- 컴포넌트 추출은 JSX가 길어서 추출하는 경우와, 실제 재사용이 있는 경우를 구분하라.
- Props의 "스펙트럼이 너무 많은" 범용 컴포넌트는 오히려 쓰는 쪽에서 더 고통스럽다.

### DRY 체크리스트

- [ ] 이 두 코드 블록이 **정말 같은 지식**인가, 아니면 우연히 닮았는가?
- [ ] 추출 후 "예외 처리용 boolean 파라미터"가 생길 가능성이 있는가? (있다면 추출 금지)
- [ ] 반복이 **3번 이상** 나타났는가? (아니면 기다려라)

## 4. SOLID 5원칙

SOLID는 객체지향(및 모듈) 설계의 5원칙이다. 이 프로젝트처럼 MVP 규모에서는 **전부 엄격히 적용하면 오버엔지니어링**이 된다. **SRP와 DIP를 먼저 엄격히 적용**하고, 나머지는 변경이 실제로 잦아질 때 도입하라.

### S — Single Responsibility (단일 책임)

**원리:** 한 모듈/함수/타입은 **변경될 이유가 단 하나**여야 한다.

**이 프로젝트에서의 적용:**
- `Handler`는 HTTP 변환만 (요청 파싱 → 서비스 호출 → 응답 직렬화).
- `Service`는 비즈니스 로직만 (bcrypt 해싱, JWT 발급, 도메인 규칙 검증).
- `Repository`는 DB 접근만 (SQL 쿼리, 로우 → 도메인 모델 매핑).
- 한 함수가 100줄을 넘으면 **거의 항상** 책임이 2개 이상이다.

**나쁜 예:**
```go
// 나쁨 - Handler가 비밀번호 해싱까지 한다 (Service의 책임을 침범)
func (h *UserHandler) Register(c *gin.Context) {
    var req RegisterRequest
    c.BindJSON(&req)
    hashed, _ := bcrypt.GenerateFromPassword([]byte(req.Password), 10)
    h.db.Exec("INSERT INTO users ...", req.Email, hashed) // DB까지 직접 건드림
}
```

**좋은 예:**
```go
// 좋음 - 각 레이어가 자기 책임만
func (h *UserHandler) Register(c *gin.Context) {
    var req RegisterRequest
    if err := c.BindJSON(&req); err != nil { /* ... */ }
    user, err := h.authService.Register(req.Email, req.Password)
    // ...
}
```

### O — Open/Closed (개방-폐쇄)

**원리:** 확장에는 열려 있고, 수정에는 닫혀 있어야 한다. 새 요구사항이 기존 코드 수정을 **강제하면 안 된다**.

**이 프로젝트에서의 적용:**
- Gin 미들웨어 체인이 좋은 예이다. 새 미들웨어(로깅, rate limiting)를 추가할 때 기존 핸들러 코드를 바꾸지 않아도 된다.
- 단, **YAGNI와 충돌**한다. 두 번째 인증 방식이 없다면 "인증 전략 패턴"을 미리 만들지 말고, 실제로 OAuth가 요구사항이 될 때 그때 추출하라.

### L — Liskov Substitution (리스코프 치환)

**원리:** 하위 타입은 상위 타입을 **예상대로** 대체할 수 있어야 한다.

**Go에서의 주의:**
- Go는 인터페이스 만족을 **구조적으로** 자동 검증한다(메서드 시그니처만 맞으면 됨). 하지만 **의미적 호환성**은 수동 체크이다.
- 예: `Repository.Save(user)`를 구현하는 두 개 중 하나가 "일부 필드를 조용히 버린다"면 구조적으로는 OK지만 의미적으로 LSP 위반이다. 호출자가 "모두 저장된다"를 가정하기 때문이다.

**실전 팁:** 인터페이스 메서드의 계약(사후조건·불변조건·에러 의미)은 **주석으로 명시**하라. "이 메서드는 email 중복 시 ErrDuplicate를 반환한다"처럼.

### I — Interface Segregation (인터페이스 분리)

**원리:** 클라이언트는 자신이 **사용하지 않는** 메서드에 의존해서는 안 된다.

**Go에서의 적용:**
- Go 표준 라이브러리의 `io.Reader`/`io.Writer`가 교과서적인 예이다. 한 메서드씩만 가진 작은 인터페이스.
- 이 프로젝트에서: `UserRepository`에 `Create/Get/Update/Delete/List/Search/Count/...` 30개 메서드를 우겨넣지 말라. 사용처에 맞춰 작게 쪼갠다:
  ```go
  type UserReader interface { GetByID(id int) (*User, error) }
  type UserWriter interface { Create(u *User) error }
  ```
- 단, **MVP 단계에서는 지나친 분리가 KISS 위반**일 수 있다. 테스트 작성 시 fake가 너무 부담스러워질 때 쪼개라.

### D — Dependency Inversion (의존성 역전)

**원리:** 상위(정책) 모듈은 하위(세부) 모듈에 의존하지 말아야 한다. **둘 다 추상에 의존하라.**

**이 프로젝트에서의 적용:**
- `Service`는 `*sql.DB`를 직접 받지 말고, `UserRepository` 타입을 받아라. 그러면 테스트 시 fake repository로 교체 가능하다.
- 현재 `backend/cmd/server/main.go`의 의존성 주입 흐름이 바로 이것이다: `db → repo → service → handler`. 각 계층이 바로 아래 계층의 "추상"에 의존한다.

**과도한 DIP는 KISS 위반:**
- 인터페이스를 3단, 5단으로 쌓지 말 것. 서비스 → 리포지토리 1단이면 충분하다.
- DI 프레임워크(wire, fx)는 MVP에서 필요 없다. 손으로 주입하는 main.go가 읽기 쉽다.

### SOLID 체크리스트 (이 프로젝트에 맞춤)

- [ ] **SRP**: 한 파일/함수가 "이 변경이 일어날 때 바뀐다"를 한 문장으로 설명 가능한가?
- [ ] **DIP**: Service가 DB 드라이버 타입을 직접 들고 있지 않은가?
- [ ] (참고) OCP/LSP/ISP는 변경이 실제로 잦아질 때 엄격히 적용. MVP에서는 SRP/DIP 우선.

---

## 5. SOLID ↔ KISS/YAGNI의 균형

SOLID는 **변경이 잦을 때 가치가 크다**. 변경이 거의 없는 코드에 SOLID를 억지로 끼워 넣으면 "미래에 있을지 모를 변경을 위한 비용"만 늘어난다 → YAGNI 위반.

**이 프로젝트의 기본 입장:**
1. SRP와 DIP는 엄격히 지킨다 (이 둘은 보수 난이도의 대부분을 결정한다).
2. OCP/LSP/ISP는 **실제로 두 번째 사례가 나타날 때** 추상화를 도입한다.
3. 한 파일이 이해하기 어려워졌다면, 거의 항상 SRP 위반이지 다른 것이 아니다. SRP부터 의심하라.

---

## 6. 이 원칙들이 충돌할 때

원칙 간 충돌은 흔하다. 우선순위는 다음과 같다:

1. **보안/가드레일** (YAGNI 예외) — 항상 최우선.
2. **SRP** — 구조를 이해 가능하게 유지.
3. **KISS** — 단순성.
4. **YAGNI** — 미래 투기 금지.
5. **DRY** (Rule of Three 이후) — 반복이 실제로 해로울 때만.
6. 나머지 SOLID — 필요할 때 도입.

충돌 시 위 우선순위로 판단하라. 특히 **DRY vs KISS**가 충돌하면 KISS 편을 들어라 — 섣부른 추상화보다 명시적 반복이 낫다.

---

## 7. 마무리 — 이 모든 원칙의 공통 목표

**"6개월 뒤의 나 자신(또는 입문자)이 이 코드를 열고 30초 안에 의도를 파악할 수 있는가?"**

이 질문에 "예"가 나오면 대부분의 원칙은 이미 지킨 것이다. "아니오"가 나오면 KISS부터 다시 검토하라.

## 7. 인프라 코드에서의 KISS / DRY 적용

### 7.1 리전 화이트리스트 가드 (KISS)

> ADR: [0007-region-strategy](../architecture/adr/0007-region-strategy.md)

허용 리전 외에서 `cdk deploy`를 실행하면 즉시 실패시킨다:

```typescript
// infra/aws-cdk/lib/shared/guards.ts
const ALLOWED_REGIONS = ['us-east-1', 'ap-northeast-1', 'ap-northeast-3'];

export function assertAllowedRegion(region: string): void {
  if (!ALLOWED_REGIONS.includes(region)) {
    throw new Error(`Region ${region} is not in whitelist: ${ALLOWED_REGIONS.join(', ')}`);
  }
}
```

이것은 KISS의 전형적 적용이다: 리전 관련 모든 복잡한 분기 로직 대신, "허용 리전이 아니면 에러"라는 단순한 가드 하나가 사고를 방지한다.

### 7.2 환경 파라미터화 (DRY)

환경별(dev/stg/prod) Stack을 복사하지 않고, 하나의 Stack에 환경 파라미터를 주입한다:

```typescript
// 나쁜 예 (DRY 위반):
new Ec2AppStackDev(app, 'Dev');
new Ec2AppStackProd(app, 'Prod');

// 좋은 예 (파라미터화):
new Ec2AppStack(app, `Ec2App-${env}`, { envConfig: loadEnvConfig(env) });
```

이 패턴의 요점: "dev와 prod의 차이는 **값**이지 **구조**가 아니다."
